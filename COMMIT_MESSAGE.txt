feat(render): Beautiful Note System -- production-quality Obsidian rendering

Replace the plain, mostly-flat Markdown templates for Algorithm, Problem,
Pattern, Mistake, and Contest with rich, skimmable Obsidian notes, while
keeping rendering fully independent of storage and the public Renderer /
Handbook APIs completely unchanged. (Topic keeps its Chunk 2 template --
it isn't part of this pass's scope.)

Reusable rendering helpers (renderers/filters.py)
- New module of small, dependency-free functions registered once as Jinja
  filters in template_engine.py: blockquote (turns any block of rendered
  Markdown into a valid Obsidian callout), editable_block (wraps prose in
  <!-- ai:*:start/end --> markers), mermaid_escape, status_emoji,
  difficulty_emoji, platform_emoji, format_dt, format_minutes.
- template_engine.render()'s signature is untouched -- filters are the
  only addition, and they're additive to the existing Environment.

Template inheritance + macros (templates/_shared/)
- layout.md.j2: every dedicated template now extends this for the one
  piece that's genuinely identical everywhere -- the closing "Metadata"
  callout (kind/id/slug/status/created/updated). frontmatter and body
  stay per-type: forcing one shape onto genuinely different fields would
  hurt readability more than sharing it would help.
- macros.j2: relation_list (bulleted relation targets with empty-state
  text), relationship_diagram (Relation lists -> Mermaid graph LR, only
  called when non-empty), metadata_footer.

Five rewritten templates (Algorithm, Problem, Pattern, Mistake, Contest)
- Consistent visual hierarchy: emoji H1 -> expanded "Quick Facts" callout
  -> expanded primary prose callout -> collapsed supplementary callouts
  (implementation/pitfalls/relationships/cross-references/sources) ->
  shared metadata footer. Every callout is `> [!type]` with `+`/`-` fold
  state chosen for progressive disclosure.
- Mermaid: each type's Relation fields (prerequisites, algorithms used,
  problems contained, ...) become a `graph LR` diagram generated straight
  from that data -- skipped entirely when there's nothing to draw, rather
  than rendering an empty graph.
- Dataview: each type emits a `dataview` query block (e.g. "Problems that
  use this algorithm", "Other mistakes in the same category", "Other
  contests on this platform"). These are plain query text resolved by
  Obsidian's Dataview plugin at view time -- no search/index/query code
  was added to the handbook itself.
- AI-managed sections: free-form prose fields (Algorithm.intuition,
  Problem/Contest.notes as Approach/Reflection, Pattern.description,
  Mistake.cause/prevention) are wrapped in <!-- ai:<field>:start/end -->
  markers -- invisible in Obsidian's reading view, a clear boundary
  around hand-authored content that's meant to persist, distinct from
  the frontmatter/relationships/footer that's fully regenerated every
  render. Storage does not yet read a file back to merge on top of these
  markers -- that's future work, not this chunk's; the markers are the
  scaffolding for it.
- Obsidian-native `aliases:` frontmatter key populated (not just a plain
  custom field), so the model's `aliases` list gives working alternate-
  name search/linking, not just decorative metadata.

Bug fixed along the way (renderers/filters.py)
- editable_block(): a whitespace-only value (e.g. "   ") is truthy in
  Python, so it was slipping past the `if value else placeholder` check
  and rendering as an empty block instead of falling back to the
  placeholder. Now checks truthiness after stripping.

A recurring Jinja gotcha, fixed everywhere it appeared
- trim_blocks silently swallows the newline after *any* line ending in a
  `{% ... %}` statement tag, even when that same line has real rendered
  content before the tag (not just when the tag is alone on its line).
  This glued adjacent callouts together and merged bulleted relation
  items onto one line. Fixed by using ternary `{{ }}` expressions instead
  of trailing `{% if %}...{% endif %}`, computing conditional text via
  `{% set %}` on its own line, and having the blockquote filter preserve
  a trailing newline like built-in Jinja filters do. A regex-based
  one-off linter caught every remaining instance across all templates
  before this landed; test_beautiful_notes.py now guards against
  regressions structurally (assert_well_formed_callouts /
  assert_no_glued_bullets on every rendered note).

Tests
- tests/test_render_filters.py (14 tests): direct unit tests for every
  filter, independent of any template.
- tests/test_beautiful_notes.py (68 tests): per-type structural checks
  across all five in-scope types, both minimal and fully-populated --
  callouts present, at least one collapsed and one expanded section,
  Mermaid present/absent correctly, Dataview blocks present, AI markers
  correctly wrap the real authored content (not the placeholder),
  frontmatter starts the file and declares `aliases:`, every note ends
  with the shared metadata footer, and no literal Python `None` leaks
  into rendered output.
- tests/test_renderer.py: updated the five Chunk 2 assertions that
  checked old header text (e.g. "# 🧠 Intuition") to match the new
  callout-based structure; added a Topic-uses-its-existing-template case
  documenting the in-scope/out-of-scope boundary explicitly.
- Full suite: 197 passed. ruff check + ruff format: clean.

Untouched (verified by diff against the Chunk 2 deliverable)
- All Pydantic models, core/storage.py, core/folders.py, core/index.py,
  core/renderer.py, handbook.py, renderers/markdown_renderer.py,
  templates/topics/topic.md.j2. No public API changed signature. Nothing
  in this chunk reads a file back before writing it.

No Search, MCP, AI read/merge workflows, graph indexing, or duplicate
detection included, per scope.
